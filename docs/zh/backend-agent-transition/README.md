# 面向后端开发者的 AI Agent 转型解读

这一组文档不是重复仓库里已有的 12 节课说明，而是站在你这样的后端开发者视角，把这个教学仓库重新拆成 4 个更适合转岗阅读的问题：

1. 这个仓库到底在教什么业务逻辑，而不只是 Python 代码？
2. 12 节课背后，系统能力是怎么一层层长出来的？
3. 如果站在产品经理角度，应该怎么理解这些机制的用户价值？
4. 如果我不熟悉 Python，能不能用 Go 的思维把里面的数据结构和伪代码读懂？

## 建议阅读顺序

1. [01-repo-business-map.md](./01-repo-business-map.md)
2. [02-agent-evolution-guide.md](./02-agent-evolution-guide.md)
3. [03-product-manager-view.md](./03-product-manager-view.md)

## 你最应该挖掘的 8 个点

1. `s01-s02` 说明了 agent 的本质不是“神秘 AI”，而是“LLM + 工具调用循环”。
2. `s03` 说明规划不是附属功能，而是可视化的执行控制。
3. `s04-s06` 说明上下文是系统资源，不是无限可用的聊天记录。
4. `s07` 说明任务状态必须离开对话，进入外部持久化。
5. `s08` 说明 agent 需要异步执行能力，不能把所有慢操作堵在主循环里。
6. `s09-s11` 说明多 agent 本质上是在做组织协作、消息总线和任务调度。
7. `s10` 说明协议、状态机、关联 ID 这些后端老问题，在 agent 系统一样存在。
8. `s12` 说明控制平面和执行平面必须分离，任务负责协调，worktree 负责隔离。

## 你可以怎么使用这组文档

- 如果你想看“全局业务”，先读第 1 篇。
- 如果你想看“从 0 到 1 的机制演进”，先读第 2 篇。
- 如果你想看“怎么从产品角度讲明白 agent 的价值”，先读第 3 篇。
- 如果你想看“怎么把 Python 代码翻译成我熟悉的后端语言脑图”，先读第 4 篇。

## 这组文档刻意不做什么

- 不研究 `web/` 目录的前端网站实现。
- 不把仓库包装成生产级框架。
- 不把每个 Python 函数逐行翻译。

目标只有一个：让你能一眼看懂这个教学仓库的业务走向、系统边界和岗位价值。
